RFQ Flow Alignment — how I understand it today

Front-end / user-perspective walkthrough of both procurement tracks: MR (Materials) and SR (Subcontractor / BOQ), from issuance through bid comparison. Built by reading the routes & components, not from memory. Written 2026-06-28.

MR = Materials track (SAP Excel, deterministic) SR = Subcontractor track (BOQ parser, flexible columns) WF = n8n webhook RPC = Supabase function DB = direct table read/write

Part 1 — Issuance (officer sends the RFQ to vendors)

The front door (both tracks start here) On the RFQ Tracker (/rfq) the + New RFQ button is a dropdown: Materials RFQ → /rfq/new · Subcontractor RFQ → /rfq/sub/new. The Tracker lists both types in one table and each row's View routes to its own detail page (/rfq/$rfqId for MR, /rfq/sub/$rfqId for SR).

MR — Materials issuance live

SAP Export → generate → invite → dispatch by email
1Generate RFQs from SAP
Officer uploads a SAP "SAPUI5 Export" .xlsx, fills optional project metadata, clicks Generate RFQs on /rfq/new.
Goes to: n8n WF7 scc-rfq-generate → creates rfqs (status draft) + rfq_items, then lands on /rfq/summary.
2Invite vendors
On /rfq/preview ("RFQ Preview & Dispatch") the officer searches the vendors table (email pulled from vendors.contacts[], blacklisted filtered out) and adds each one.
Goes to: DB insert into rfq_vendors with email_to, status pending. No token yet.
3Edit covering email & dispatch
How you get to dispatch: from the detail page Vendors tab (while draft) → set deadline → Preview & Select Recipients → /rfq/preview. (Also reachable from the post-generate summary page.)
There the officer edits covering-email subject/body, ticks the vendors to send to, clicks Send RFQ to N Vendors (confirmation modal with deadline).
Goes to: saves email to rfqs, then n8n WF8 scc-rfq-dispatch with the selected vendor ids + deadline.
WF8 does (backend): stamps sent_at, flips rfqs.status → issued, sends the covering email. (Note: bid_token is not minted here — it already exists on the vendor rows from insert, even while draft. Verified via DB 2026-06-28.)
4Manual link fallback
On the detail page Vendors tab, BidLinksPanel lists every dispatched vendor with copy / mailto buttons for their /bid/<token> link — used until automated email is trusted, or to re-send.
5Vendor quotes
Vendor opens /bid/<token> → RPC bid_get_by_token, enters unit rates + commercial terms, clicks Submit quotation → RPC bid_submit_by_token (revisable until deadline).

SR — Subcontractor issuance live (testing)

BOQ upload → parse → curate columns → issue
1Open the SR RFQ & the Issue tab
Officer is on the subcontractor RFQ page /rfq/sub/<rfqId> and clicks the Issue BOQ tab. Panel checks: is an sr_boq already issued? loads rfq_vendors, pings parser health.
2Upload & parse the BOQ
Drag/drop a PDF or .xlsx BOQ → parseBoqRemote calls the external parser at VITE_BOQ_SERVICE_URL (auth X-BOQ-Key, ~20–60s vision model). Returns flexible columns + rows + parser notes + scope.
3Curate what vendors see
Officer eye-toggles columns (price/budget cols auto-hidden), edits any cell, fixes flagged/incomplete rows, sets the Scope line. A red price-leak banner warns if numbers sit in a hidden column.
4Issue the BOQ
Click Issue BOQ to vendors → RPC sr_boq_issue builds the sr_boq + sr_boq_line skeleton. Internal columns are stored but projected out server-side so they never reach a vendor.
5Hand out per-vendor links
Panel flips to the issued state and lists each invited vendor with their /sr-bid/<token> link (copy / open). Token comes from rfq_vendors.bid_token.
6Vendor quotes
Vendor opens /sr-bid/<token> → RPC sr_bid_get_by_token (visible columns only), enters unit rates + remarks + commercial terms, submits → RPC sr_bid_submit_by_token (sr_bid/sr_bid_line, revisions, deadline lock, reopen window).
⚠ The one real coupling to confirm: how do SR vendors get added? The SR Issue panel only reads rfq_vendors.bid_token — it never creates it. The token itself isn't the problem (it's generated at row-insert, draft-time — verified via DB). The open question is the vendor-invite step: SR vendor rows still come from the materials-style invite path (the same rfq_vendors insert used by the Preview page). So SR doesn't yet have its own clean "invite these subcontractors" entry — it leans on the materials machinery. That's the biggest "are we on the same page?" item — see the page-overlap note below.
The SR page mixes three "send to vendor" mechanisms today /rfq/sub/<rfqId> has 5 tabs: Overview (materials-style email dispatch panel), Vendors (materials BidLinksPanel + invite), Documents (Drive upload via n8n), Issue BOQ (the new SR parser flow), and BoQ Upload (the older "frame" path → scc-frame-generate → rfq_items). Three of those are different ways to push work to a vendor. Worth deciding which is the canonical SR spine before we build the SR comparison sheet.
Two bid-collection models — and the decision to converge them Model A (MR today): the dispatched email contains the item schedule as a table and says "reply directly to this email" — vendors email their prices back and n8n + AI extraction turns the reply into a bid (that's what the Bid-Review "Original Email | AI-Extracted Data" screen is built for). Model B (SR, and MR's manual fallback): the vendor gets a private /bid/<token> / /sr-bid/<token> link and submits on a web form — no email parsing.

Decision 2026-06-28: MR moves to Model B (link-first), like SR. Load-bearing change is backend — bid_token is minted by WF8 at dispatch, so only WF8 can embed each vendor's link; the WF7/WF8 email template must carry the portal link instead of the table. Frontend portal + BidLinksPanel already exist, so the model is testable today via manual link hand-out. Likely FE follow-up: demote the "Original Email" panel on Bid Review for portal-submitted bids.

Part 2 — Bid comparison & award

MR — Materials comparison live

review each bid → matrix → recommend → award → approve → PO
1Review each bid
Per bid on /rfq/<rfqId>/bids/<bidId>/review: left = original vendor email, right = AI-extracted fields + line rates (colour-coded by confidence), all editable. Officer clicks CONFIRM BID (→ status confirmed), or Save Draft / Reject.
2Comparison matrix
On /rfq/<rfqId>/comparison: items × vendors grid, unit rate + amount per cell, lowest highlighted green, missing = NQ, plus Budget Rate/Amt and a Min column; footer subtotal / VAT / total. A Commercial Summary table compares payment terms, lead time, validity, brand, vendor status.
Data: DB bids + bid_items joined to rfq_items, ordered by total.
3AI recommendation
Generate AI Recommendation → n8n WF11 scc-rfq-recommendation writes a pick + reasoning to comparisons.
4Per-line award + equalization
ComparisonAwardPanel awards each line to a vendor and applies equalization adjustments; "Mark as Final" / Decision card capture approved column + lowest-vs-not-lowest + notes.
5Approval → PO
Submit for approval mints a single-use /comparison-review/<token> link for Rabia. On approval, officer enters the PO number → RPC comparison_issue_po → status po_issued (locked).

Part 4 — Design philosophy (agreed 2026-06-28)

Principles 1. Lifecycle-first IA — structure mirrors the journey (Define → Solicit → Collect → Evaluate → Award → Approve → PO) with a persistent status stepper that always shows "what stage, what's next."
2. One mental model across MR & SR — same page skeleton; type differences (SAP vs BOQ; email vs portal) live inside a stage.
3. Separate what / who / responses / award — content (items+email+T&Cs) ≠ recipients ≠ bids ≠ evaluation. Email leaves the Vendors tab.
4. One canonical action per decision — no two buttons writing the same state (kills "Mark as Final" vs "Submit for approval", and single-column Decision card vs per-line Award).
5. Progressive disclosure gated by stage — only valid actions are shown for the current stage.
6. Formalize the link/token seam — every shareable link (bid, approval) has a state: manual-copy → (later) auto-emailed, so n8n automation drops in with no UI rework.
7. Safe, additive migration — introduced alongside current pages; nothing existing breaks (per CLAUDE.md).
Agreed forks Hybrid navigation (stepper + jumpable tabs) · heavy views (comparison, dispatch) stay dedicated full-width routes framed as stages · MR first, SR adopts the same skeleton after.
Ingestion is a pluggable adapter, not a stage (future-proofs MR→LLM) The canonical artifact is the items table (rfq_items MR / sr_boq_line SR); the source is a swappable front-end on the Define/"RFQ Document" stage — today SAP→WF7 (MR, deterministic) and BOQ→parser→HITL (SR). When MR later adopts the Python/LLM parser for ingestion it's just another adapter feeding the same table: no new stage, no stepper change, no schema change (LLM fills rfq_items like WF7). So SR's parse→curate→confidence panel is built as a shared component, and HITL/confidence states are generic (the SAP path simply doesn't trigger them). LLM is already in MR on the bid side (Model-A email→AI extraction). Future caveat: MR ingesting arbitrary unstructured docs may eventually want SR's flexible-column model — that would be backend; not a blocker now, IA kept able to flex toward it.
