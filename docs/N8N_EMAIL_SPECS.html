n8n email automation specs — for the backend operator

The only backend work left for the procurement system. All three are email automations in n8n. The frontend already mints every token/link and shows it for manual copy as a fallback — these automations just send the email. Written 2026-06-28.

⚠ Back up each workflow BEFORE editing (do this first) In the n8n UI, open the workflow → top-right ⋯ menu → Download (exports the workflow JSON to a file) — keep that file as the rollback. Alternatively Duplicate the workflow and edit the copy, switching the webhook over only once tested. Never edit a live workflow without an exported copy. (These specs were authored from the frontend repo, which cannot read or edit n8n — the connector is read-only + currently expired — so the operator applies them.)
Integration pattern (matches the existing WF7/WF8) Easiest + consistent with how the app already works: the frontend POSTs to an n8n webhook after the relevant action, passing the IDs + the already-minted link. n8n just composes + sends the email. The frontend changes are tiny and can be added on request (fire-and-forget, so a missing webhook never breaks the action). Recipient addresses should come from a system_settings row (e.g. approver_email) rather than hard-coding — there is already a system_settings table.

On "Submit for approval" → email Rabia the single-use review link

Replaces the current manual copy/paste of the review link. Confirmed needed (handoff #6).

  "rfq_reference": "MR-2606-035",
  "title": "Building Materials …",
  "review_url": "https://<app>/comparison-review/<token>",   // or /sr-comparison-review/<token>
  "prepared_by": "younus@zavia-ai.com"
}
RecipientApprover email from system_settings.approver_email (add this row; current approver = Rabia, rabia.vahabudeen@sarooj.com). Honour dispatch_test_mode like WF8 (in test mode send to the test recipients instead).
Email subjectApproval needed — {rfq_reference} comparison
Email bodyShort note + the review_url as the single call-to-action: "A procurement comparison for {rfq_reference} — {title} is ready for your approval. Review & decide here: {review_url}. Prepared by {prepared_by}." The link lets her Approve / Return / Revoke; it stays live until the PO is issued.
Frontend change (small, FE-owned)After a successful submit, POST the payload to the webhook (fire-and-forget). MR: rfq.$rfqId.comparison.tsx handleSubmitForApproval. SR: sr-comparison-panel.tsx handleSubmitApproval. Both already have the token in the RPC response.

2 — Notifications MR + SR

(a) Notify procurement when a vendor submits a quotation

So the officer knows a bid arrived without polling.
TriggerWebhook /webhook/scc-bid-submitted, OR (cleaner) a Supabase DB trigger / scheduled poll on bids (materials) and sr_bid (subcontractor) for new is_latest rows. Webhook is simplest: the vendor submit RPCs already run server-side — the frontend bid/sr-bid pages can POST on success.
Payload
{ "kind":"materials"|"subcontractor", "rfq_reference":"…",
  "vendor_name":"…", "total_omr": 6783 }
RecipientProcurement officer(s) — a system_settings.officer_emails list (or the RFQ's created_by).
Subject / bodyNew quotation — {rfq_reference} from {vendor_name} · body links to the RFQ's Bids tab.

(b) Notify the officer on approve / return / revoke

TriggerInside the same workflow as comparison_decide_by_token / sr_comparison_decide_by_token — or a webhook /webhook/scc-approval-decided the decide step calls. Since the decision happens in a SECURITY DEFINER RPC (no n8n today), the cleanest is a Supabase DB trigger on comparisons.status / sr_comparison.status change → call n8n. If DB triggers aren't desired, add an n8n webhook the approver review page calls after a decision.
RecipientThe prepared_by officer on the comparison.
Subject{rfq_reference} — comparison {approved | returned | revoked} · body includes the approver's note (review_notes) on return/revoke.
